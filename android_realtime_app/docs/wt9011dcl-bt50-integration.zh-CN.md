# WT9011DCL-BT50 Android 集成笔记

本文是面向本 Android App 的 WT9011DCL-BT50 开发摘要，来源于本地物理实现资料、厂商通信协议/规格文档，以及当前仓库代码。目标是帮助后续开发者快速理解硬件数据如何进入 App，而不是替代完整厂商手册。

## 1. 设备与本项目的关系

WT9011DCL-BT50 是 BLE 5.0 惯性姿态传感器。对本项目最关键的数据是：

- 三轴加速度：`accX`, `accY`, `accZ`
- 三轴角速度：`gyroX`, `gyroY`, `gyroZ`

当前识别模型假设输入为 100 Hz、6 通道、物理单位数据，通道顺序为：

```text
accX, accY, accZ, gyroX, gyroY, gyroZ
```

角度和磁场主要用于姿态展示、方向估计和辅助校准，不进入当前 `MotionClassifier` 的模型输入。

## 2. 采购可得性

WT9011DCL-BT50 属于比较容易采购的现成 BLE IMU 模块。开源复现时，通常可以直接在淘宝/天猫、京东、Amazon Marketplace、eBay 等常见电商平台搜索 `WT9011DCL-BT50` 或 `WitMotion WT9011DCL-BT50` 购买。

不同地区的平台库存、卖家、价格、物流和随附配件会变化；复现实验时应确认购买的是 `WT9011DCL-BT50`，并留意是否包含绑带、蓝牙适配器、数据线或其他附件。

## 3. BLE 连接参数

当前 Android 实现使用以下固定 UUID：

| 用途 | UUID | 当前 App 状态 |
| --- | --- | --- |
| Service | `0000ffe5-0000-1000-8000-00805f9a34fb` | 已使用 |
| Notify/Read | `0000ffe4-0000-1000-8000-00805f9a34fb` | 已订阅 |
| Write | `0000ffe9-0000-1000-8000-00805f9a34fb` | 已定义，当前未暴露写命令流程 |
| CCC descriptor | `00002902-0000-1000-8000-00805f9b34fb` | 已用于开启通知 |

本项目的连接流程：

1. 扫描 BLE 设备。
2. 选择名称包含 `WT` 的设备。
3. 连接 GATT。
4. 获取 `FFE5` 服务。
5. 订阅 `FFE4` 特征通知。
6. 将通知字节流解析为 `ImuData`。

对应实现：

- `app/src/main/java/com/imu/realtime/BleManager.kt`
- `tools/desktop/collect.py`
- `tools/desktop/server.py`

## 4. 默认 IMU 数据帧：`55 61`

设备默认输出 `0x55 0x61` 开头的 20 字节数据帧。本 App 主要依赖这一帧。

### 4.1 字节布局

| 字节范围 | 含义 | 说明 |
| --- | --- | --- |
| `0` | `0x55` | 帧头 |
| `1` | `0x61` | 标志位 |
| `2..3` | `AccX` | 小端 signed int16 |
| `4..5` | `AccY` | 小端 signed int16 |
| `6..7` | `AccZ` | 小端 signed int16 |
| `8..9` | `GyroX` | 小端 signed int16 |
| `10..11` | `GyroY` | 小端 signed int16 |
| `12..13` | `GyroZ` | 小端 signed int16 |
| `14..15` | `AngleX` | 小端 signed int16 |
| `16..17` | `AngleY` | 小端 signed int16 |
| `18..19` | `AngleZ` | 小端 signed int16 |

本 App 按 20 字节收包，当前没有做校验和验证。

### 4.2 换算规则

先按小端组合为 signed int16：

```text
raw = signed_int16(low_byte, high_byte)
```

再换算为物理量：

| 数据 | 公式 | 单位 |
| --- | --- | --- |
| 加速度 | `raw / 32768 * 16` | `g` |
| 角速度 | `raw / 32768 * 2000` | `deg/s` |
| 欧拉角 | `raw / 32768 * 180` | `deg` |

当前 Kotlin 实现见 `BleManager.parseFrame61()`。

## 5. 寄存器读取与扩展数据

BLE 写通道本质上承载的是厂商寄存器命令。写入 `FFE9` 的命令通常仍是 `FF AA ...` 格式。

### 5.1 读寄存器

格式：

```text
FF AA 27 REG 00
```

常见命令：

| 目标 | 命令 |
| --- | --- |
| 磁场 | `FF AA 27 3A 00` |
| 四元数 | `FF AA 27 51 00` |
| 温度 | `FF AA 27 40 00` |
| 电量 | `FF AA 27 64 00` |
| 版本号 | `FF AA 27 2E 00`, `FF AA 27 2F 00` |

厂商协议中，读寄存器返回帧以 `55 71` 开头，包含起始寄存器和后续 8 个寄存器的数据。

### 5.2 磁场与当前 App

厂商寄存器读取路径使用 `55 71` 返回磁场数据。当前 Android `BleManager` 还接受 `55 54` 磁场帧，并把磁场合并到上一帧 `acc/gyro/angle` 数据里。后续如果要统一协议实现，应先确认目标固件实际返回的是 `55 54` 主动帧、`55 71` 寄存器返回帧，还是两者都可能出现。

## 6. 常用配置命令

修改配置的一般流程：

1. 解锁。
2. 发送具体配置命令。
3. 需要持久化时保存配置。

### 6.1 解锁和保存

| 操作 | 命令 |
| --- | --- |
| 解锁寄存器 | `FF AA 69 88 B5` |
| 保存当前配置 | `FF AA 00 00 00` |
| 恢复默认并保存 | `FF AA 00 01 00` |

厂商说明中要求修改命令应在解锁后约 10 秒内完成。

### 6.2 设置回传速率

格式：

```text
FF AA 03 RATE 00
```

常见 `RATE` 值：

| 值 | 频率 |
| --- | --- |
| `0x06` | 10 Hz，常见默认值 |
| `0x07` | 20 Hz |
| `0x08` | 50 Hz |
| `0x09` | 100 Hz |
| `0x0B` | 200 Hz |

本 App 的识别管线和模型卡按 100 Hz 设计。实际使用前应确认顶部状态栏采样率接近 `100 Hz`；如果设备仍是默认 `10 Hz`，3s/5s/8s 窗口会变成错误的物理时长。

当前 App 没有主动向 `FFE9` 写入设置回传速率命令。需要 100 Hz 时，可先用厂商工具或后续在 App 中补充写命令流程：

```text
FF AA 69 88 B5
FF AA 03 09 00
FF AA 00 00 00
```

### 6.3 设置带宽

格式：

```text
FF AA 1F BAND 00
```

常见 `BAND` 值：

| 值 | 带宽 |
| --- | --- |
| `0x00` | 256 Hz |
| `0x01` | 188 Hz |
| `0x02` | 98 Hz |
| `0x03` | 42 Hz |
| `0x04` | 20 Hz，常见默认值 |
| `0x05` | 10 Hz |
| `0x06` | 5 Hz |

运动识别更依赖动态信息。若回传速率提升到 100 Hz，但带宽仍过低，可能出现波形过度平滑或重复数据。实际选择需要结合采样稳定性、噪声和模型训练条件。

### 6.4 校准

| 操作 | 命令 |
| --- | --- |
| 加速度校准 | `FF AA 01 01 00` |
| 开始磁场校准 | `FF AA 01 07 00` |
| 结束磁场校准 | `FF AA 01 00 00` |
| 设置角度参考 | `FF AA 01 08 00` |
| Z 轴角归零 | `FF AA 01 04 00` |

磁场相关功能对环境更敏感。厂商规格资料建议远离强磁、电子设备、磁铁、扬声器等干扰源。当前动作识别模型只使用 `acc + gyro`，通常不需要先完成磁场路径。

## 7. App 侧实现关系

### 7.1 实时数据流

```text
BLE notify bytes
-> BleManager.parseBytes()
-> ImuData
-> ImuViewModel.onData()
-> charts / attitude / hand / trajectory / dashboard / recognition
```

`ImuViewModel.onData()` 会同时做三件事：

- 发布最新 `ImuData` 给 UI。
- 交给 `DataRecorder` 写 CSV。
- 取 `accX..gyroZ` 六通道送入 `MotionClassifier.addSample()`。

### 7.2 识别模型输入

`MotionClassifier` 累积完整 session 数据，每约 100 个样本触发一次重新推理。按 100 Hz 设计时，这相当于约 1 秒更新一次。

模型窗口：

| 模型 | 样本数 | 物理时长，按 100 Hz |
| --- | --- | --- |
| 3s | 300 | 3 秒 |
| 5s | 500 | 5 秒 |
| 8s | 800 | 8 秒 |

如果实际回传频率偏离 100 Hz，需要重新审视窗口大小、步长和模型训练条件。

### 7.3 CSV 录制

当前 CSV 字段：

```text
timestamp_ms,accX,accY,accZ,gyroX,gyroY,gyroZ,angleX,angleY,angleZ,magX,magY,magZ
```

保存位置是 Android Downloads 目录，文件名形如：

```text
imu_yyyyMMdd_HHmmss.csv
```

## 8. 开发检查清单

接入或调试新设备时，建议按下面顺序确认：

1. 设备名称能被扫描到，且名称中包含 `WT`。
2. GATT 服务 `FFE5` 存在。
3. `FFE4` 通知能打开。
4. `55 61` 帧持续到达，帧长按 20 字节解析。
5. 顶部采样率接近期望值，运动识别场景应接近 `100 Hz`。
6. 静置时加速度模长大致接近 `1 g`。
7. 转动设备时角速度和角度有符合直觉的变化。
8. 识别页缓冲进度能到达 `800/800`。
9. 若要稳定复现实验，记录设备佩戴位置、方向、回传速率、带宽和校准状态。
